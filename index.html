Kakfa实践：二、生产者

Posted on 2020-01-25 | In Kafka

二、生产者

讲述Kafka生产者的基本使用，包括如何配置一个Kafka客户端，发送消息的方式以及客户端的订制开发，随后进行客户端原理剖析。

1.基本使用

1.1 客户端配置必要的参数

  • bootstrap.servers:Kafka集群的broker的地址清单，以“**,**”分隔，只需要配置部分broker地址，其他地址可通过自动发现自动查找。
  • key.serializer:key的序列化器
  • value.serializer:value的序列化器

所谓的序列化简单来说就是把Java的对象或者基本类型转换为byte[]，用于网络传输。

1.2 消息发送的方式

  • 发后即忘：只负责发送，不关注发送结果。
  • 同步：必须等待消息发送完成并获取结果。

注意：send()方法本身是异步的，它返回一个Future<>对象，这是Java提供的异步获取结果的方式，但是get()方法是同步的。Future的设计理念是多线程方法立刻返回一个Future的对象，但是这个对象的get()方法应该在线程真正返回的时候调用，否则会导致线程挂起等待直到结果返回，可以通过isDone()判断是否计算完，cancel()取消计算，isCancelled()判断是否被取消。

  • 异步：通过客户端提供的Callback()接口实现，RecordMetadata和Exception互斥，只有一个非null。

1.3 发送的过程

消息按序经过拦截器，序列化器，分区器。

1.3.1 拦截器

  1. public ProducerRecord<K, V> onSend(ProducerRecord<K, V> record)方法：此方法会在序列化之前调用，可以用来对发送消息做记录或者修改（不推荐）。

  2. void onAcknowledgement(RecordMetadata metadata, Exception exception)方法：该方法确保一定会被调用，具体调用顺序如下：

    • 在发送成功，服务器响应之后，callback()方法调用之前被调用；
    • 在发送失败之后调用。

  3. void close()方法：用于清理资源，一般不需要重写，重写请注意幂等性。

    1.3.2 序列化器

  4. byte[] serialize(String topic, T data)方法：用于序列化和反序列化。

    1.3.3 分区器

  5. public int partition(String topic, Object key, byte[] keyBytes, Object value, byte[] valueBytes, Cluster cluster)方法：用于分区

    2.原理剖析

    2.1 消息累加器

    网络请求是需要消耗带宽的，消息累加器负责把发往同一分区的消息批处理，·以节省网络带宽。也用于复用ByteBuffer，通过维护BufferPool实现，但只有数据<=batch.size才会被复用。

    2.2 InFlightRequest

    负责把消息按broker再次归类，并缓存发送过的消息，方便在发送失败时重试，数量为max.in.flight.requests.per.connection，注意，如果要求消息有序，请将该值设为1。

    2.3 元数据

    元数据存储了Kafka集群的信息，会由客户端定时更新，时间间隔为metadata.max.age.ms。

Kakfa实践：一、基本概念

Posted on 2020-01-15 | In Kafka

一、基础概念

介绍Kafka的基本组成和常用概念。

1.Producer

生产者，负责创建消息，并将消息发送到Kafka。

2.Consumer

消费者，负责连接到Kafka并接收消息。

3.Broker

服务代理节点，可以看作一个Kafka服务器实例，一个或多个Broker组成一个Kafka集群。

3.1 Topic

主题，消息发送和接收的基本单位，对消息归类形成主题，生产者生产消息时需要指定主题，消费者消费时同样需要指定主题。

3.2 Partition

分区，一个主题有多个分区，一个分区只属于一个主题，同一主题的不同分区包含不同的消息。消息在发送到broker时会根据规则选择分区。

3.2.1 Replicas

分区副本，一主多从关系，主副本为Leader，负责处理读写请求；从副本为Follower，只负责同步Leader消息，不处理读写请求；当Leader故障时，会从Follower中按规则选举出一个晋升为Leader。
Kafka中所有副本统称为AR（Assigned Replicas）。

3.2.1.1 ISR（In-Sync Replicas）

所有与Leader副本保持一定程度同步的副本（包括Leader副本）。

3.2.1.2 OSR（Out-Of-Sync Replicas）

所有与Leader副本同步滞后的副本（不包括Leader副本）。

3.2.2 Offset

消息在分区内的唯一标识，不可跨分区。

3.2.2.1 HW（High Watermark）

一个特定的偏移量，消费者只能拉到该偏移量之前的消息[LSO,HW)。[HW,LEO)是未同步完成的偏移量。

3.2.2.2 LSO（LogStartOffsert）

起始偏移量。

3.2.2.3 LEO（LogEndOffsert）

当前将要写入的最新消息的偏移量。
